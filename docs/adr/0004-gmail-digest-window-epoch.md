# ADR-0004: gmail-digest の集計期間を前日7:00〜当日7:00のepoch境界に変更する

- Status: Accepted
- Date: 2026-06-21

## Context

ADR-0002 では gmail-digest の対象期間を「**前日の暦日（JST 0:00〜23:59）**」と定め、`after:`/`before:` に `yyyy/MM/dd` を渡す**日付粒度**のクエリとした。理由は「時刻での絞り込みをせず実装をシンプルに保つ」ためとした。

実装を見直した結果、この前提が崩れていた。

- `getYesterdayWindow` は JST 変換のオフセット計算を手の込んだ形で行っているのに、クエリに渡すのは結局 `after:2026/06/20 before:2026/06/21` という日付だけで、「時刻を切るから簡潔」という説明と実装が噛み合っていない。
- トリガーは**当日7時台**に発火する。対象を前日暦日（〜23:59）に固定すると、**当日0:00〜発火時刻**に届いた分が当日の集計に入らない。翌日の実行で拾われるため消失はしないが、ウィンドウが実行タイミングと噛み合っておらず、1日遅れて報告される。
- GAS の時間トリガーは「7時〜8時」のように幅を持って発火し、**具体的な発火時刻は不定**。

## Decision

### 集計期間を「前日7:00〜当日7:00（JST）」の24時間ローリングに変更

- 対象期間を `[前日 07:00, 当日 07:00)`（当日 07:00 は排他境界＝「当日6:59まで」）とする。発火しうる最も早い時刻である 7:00 を境界に固定する。
- これにより隣接する実行が `[D-1 07:00, D 07:00)` → `[D 07:00, D+1 07:00)` と**隙間なくタイル状に連続**する。トリガーが 7:15 でも 7:45 でも、取りこぼし・重複が出ない（7:00 以降に届いた分は翌日のウィンドウへ回る）。

### 時刻境界は epoch 秒で表現する

- Gmail の `after:`/`before:` は **Unix タイムスタンプ秒**を受け付けるため、日付ではなく epoch 秒で境界を渡す。クエリは `label:newsletter after:<前日07:00のepoch> before:<当日07:00のepoch>`。
- 日付粒度（`yyyy/MM/dd`）では 7:00 境界を表現できないため、epoch 化は本変更に必須。

### 関数・戻り値の再設計

- `getYesterdayWindow` を **`getDigestWindow`** に改名する（型 `YesterdayWindow` → `DigestWindow`）。実態が「前日暦日」ではなくなったため。
- 戻り値を `{ after: number; before: number; dateLabel: string }` に分離する。`after`/`before` はクエリ用の epoch 秒、`dateLabel` は Slack 表示用の前日日付（`yyyy/MM/dd`）。クエリ境界と表示日付の関心を分ける。

## Consequences

- ADR-0002 の「対象は前日の暦日／暦日単位とし、時刻での絞り込みはしない」は本ADRで**更新**される。CONTEXT.md の用語「前日ウィンドウ」の定義も前日7:00〜当日7:00へ更新した。
- クエリが日付から epoch 秒に変わり、`getDigestWindow` は表示用ラベルと境界 epoch を別々に返すようになる。
- ウィンドウが実行サイクルと一致し、当日早朝の到着分が1日遅れる問題が解消する。
- 「前日ウィンドウ」という呼称は前日 7:00 起点の概念名として残す（暦日という意味ではない）。
